## 1. 错误处理原则

### 1.1 统一性原则
- 所有错误必须通过统一的错误处理机制处理
- 错误消息格式统一，用户友好
- 错误日志记录规范统一

### 1.2 用户体验原则
- 错误消息清晰明确，用户可理解
- 避免显示技术性错误信息给用户
- 提供明确的错误解决建议

### 1.3 可维护性原则
- 错误码统一管理
- 错误处理逻辑集中管理
- 便于错误追踪和调试

## 2. 错误分类

### 2.1 网络错误（Network Error）
- **特征**：请求未到达服务器
- **常见情况**：
  - 网络连接失败
  - 请求超时
  - DNS解析失败
- **处理方式**：提示用户检查网络连接

### 2.2 HTTP错误（HTTP Error）
- **特征**：请求到达服务器，但返回非2xx状态码
- **常见状态码**：
  - 400: 请求参数错误
  - 401: 未授权
  - 403: 无权限
  - 404: 资源不存在
  - 500: 服务器错误
- **处理方式**：根据状态码显示对应错误消息

### 2.3 业务错误（Business Error）
- **特征**：HTTP状态码200，但业务逻辑失败
- **表现形式**：`code !== 200`
- **处理方式**：显示业务错误消息

### 2.4 验证错误（Validation Error）
- **特征**：参数验证失败
- **表现形式**：HTTP 422 或业务错误码 1200-1299
- **处理方式**：显示详细的字段验证错误

## 3. 错误响应格式

### 3.1 标准错误响应
```json
{
  "code": 400,
  "msg": "错误描述信息",
  "data": null,
  "timestamp": 1704787200000
}
```

### 3.2 验证错误响应
```json
{
  "code": 422,
  "msg": "参数验证失败",
  "data": {
    "username": ["用户名不能为空", "用户名长度必须在2-50之间"],
    "email": ["邮箱格式不正确"]
  },
  "timestamp": 1704787200000
}
```

## 4. 错误码规范

### 4.1 HTTP状态码
遵循标准HTTP状态码规范，详见API接口文档。

### 4.2 业务错误码
采用分段编码方式，便于管理和扩展：

- **1000-1099**: 认证相关错误
- **1100-1199**: 权限相关错误
- **1200-1299**: 参数验证相关错误
- **1300-1399**: 数据相关错误
- **1400-1499**: 业务逻辑相关错误
- **1500-1599**: 系统相关错误

### 4.3 错误码映射
前端维护错误码到错误消息的映射表，详见 `utils/errorHandler.js`。

## 5. 前端错误处理

### 5.1 统一错误处理工具
使用 `utils/errorHandler.js` 统一处理所有错误。

### 5.2 使用方式

#### 5.2.1 在API请求中使用
```javascript
import request from '@/utils/request'

// 自动处理错误（推荐）
export function getUserList(params) {
  return request({
    url: '/user/list',
    method: 'get',
    params
  })
}
```

#### 5.2.2 手动处理错误
```javascript
import ErrorHandler from '@/utils/errorHandler'

try {
  const result = await getUserList(params)
  // 处理成功结果
} catch (error) {
  // 自定义错误处理
  ErrorHandler.handle(error, {
    showMessage: true,  // 是否显示错误消息
    logError: true,     // 是否记录错误日志
    onError: (err) => { // 自定义处理回调
      console.log('自定义错误处理', err)
    }
  })
}
```

#### 5.2.3 静默处理错误
```javascript
try {
  const result = await getUserList(params)
} catch (error) {
  // 不显示错误消息，只记录日志
  ErrorHandler.handle(error, {
    showMessage: false,
    logError: true
  })
}
```

### 5.3 特殊错误处理

#### 5.3.1 401未授权错误
- 自动清除本地token和用户信息
- 显示登录过期提示
- 自动跳转到登录页

#### 5.3.2 403无权限错误
- 显示权限不足提示
- 不自动跳转（由业务逻辑决定）

#### 5.3.3 422验证错误
- 显示详细的字段验证错误
- 支持多个字段的错误消息

## 6. 后端错误处理

### 6.1 统一响应格式
所有接口必须返回统一的响应格式：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": {},
  "timestamp": 1704787200000
}
```

### 6.2 错误响应规范
- **业务错误**：HTTP状态码200，`code`字段表示错误码
- **HTTP错误**：使用标准HTTP状态码（400, 401, 403, 404, 500等）
- **验证错误**：HTTP状态码422，`data`字段包含字段验证错误

### 6.3 错误消息规范
- 错误消息必须清晰明确
- 避免显示技术性错误信息
- 提供用户可理解的错误描述

---

## 7. 错误日志

### 7.1 日志记录原则
- 开发环境：记录详细错误信息（包括堆栈）
- 生产环境：记录关键错误信息，避免敏感信息泄露

### 7.2 日志内容
- 错误类型和错误码
- 错误消息
- 请求URL和参数
- 响应数据
- 时间戳

### 7.3 日志上报（可选）
生产环境可以将错误日志上报到日志收集服务，便于问题追踪和分析。

## 8. 最佳实践

### 8.1 前端最佳实践

1. **使用统一的错误处理工具**
   ```javascript
   // ✅ 推荐：使用统一工具
   import ErrorHandler from '@/utils/errorHandler'
   ErrorHandler.handle(error)
   
   // ❌ 不推荐：直接使用ElMessage
   ElMessage.error('操作失败')
   ```

2. **合理使用try-catch**
   ```javascript
   // ✅ 推荐：在需要的地方捕获错误
   try {
     await submitForm()
     ElMessage.success('提交成功')
   } catch (error) {
     // 错误已在拦截器中处理，这里可以做额外处理
   }
   ```

3. **避免重复显示错误消息**
   ```javascript
   // ✅ 推荐：让拦截器统一处理
   const result = await api.getData()
   
   // ❌ 不推荐：重复显示错误
   try {
     const result = await api.getData()
   } catch (error) {
     ElMessage.error('获取数据失败') // 拦截器已经显示过了
   }
   ```

### 8.2 后端最佳实践

1. **统一异常处理**
   - 使用全局异常处理器
   - 统一异常响应格式

2. **错误码管理**
   - 使用枚举类管理错误码
   - 错误码与错误消息分离

3. **验证错误处理**
   - 使用统一的参数验证框架
   - 返回详细的字段验证错误

## 9. 错误处理流程图

```
请求发送
  ↓
请求拦截器（添加token等）
  ↓
发送请求
  ↓
响应拦截器
  ↓
判断响应状态
  ├─ 成功（code === 200）→ 返回数据
  ├─ 业务错误（code !== 200）→ 错误处理
  └─ HTTP错误（非2xx）→ 错误处理
  ↓
错误处理工具
  ├─ 解析错误类型
  ├─ 映射错误消息
  ├─ 记录错误日志
  ├─ 显示错误消息
  └─ 特殊错误处理（401跳转等）
```

## 11. 常见问题

### Q1: 如何自定义错误处理？
A: 使用ErrorHandler.handle的onError回调参数。

### Q2: 如何静默处理错误（不显示消息）？
A: 设置 `showMessage: false`。

### Q3: 如何获取详细的验证错误信息？
A: 错误对象的 `validationErrors` 属性包含字段验证错误。

### Q4: 如何添加新的错误码？
A: 在 `utils/errorHandler.js` 的 `BUSINESS_ERROR_MAP` 中添加映射。
